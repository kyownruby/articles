# Gitって何だろう？ ── 怖くない、はじめの一歩

## はじめに

ちょっと考えすぎかもしれないんですが…🤭

「Git（ギット）」って名前、
プログラミングをしていなくても、
なんとなく聞いたことがある方もいるかもしれません。

でもいざ調べてみると、
黒い画面（「ターミナル」や「コマンドプロンプト」と呼ばれる、文字だけで操作する画面のことです）に
英語のコマンドを打ち込んでいて、
「コミット」「ブランチ」「マージ」……と
聞き慣れない言葉がたくさん出てくる。

正直、最初はちょっと怖かったです💦

でも実は、
Gitがやっていることって、すごくシンプルなんですよね🌸

今日はそのあたりを、
プログラミングの経験がなくてもわかるように、
一緒にゆっくり整理してみたいなと思います😇


## そもそも、Gitって何をしてくれるの？

少しだけ真面目な話をするとですね🤭

Gitは「**バージョン管理システム**」と呼ばれるものです。

……って言われても、ピンとこないですよね💦

もう少しやわらかく言うと、
**ファイルの「変更の歴史」を自動で記録してくれるしくみ** です✨

たとえば、こんな経験ないでしょうか。

- レポートを書いていて、「やっぱり昨日の版に戻したい」と思った
- グループワークで同じファイルを編集していたら、誰かの変更が消えてしまった
- `レポート_最終版_本当の最終版_v3.docx` みたいなファイル名が増えていった

Gitを使うと、
こういった悩みから解放されます🌸

いつでも過去の状態に戻れるし、
誰がいつ何を変えたかも全部わかる。

イメージとしては、
**「セーブポイントを自由に作れるゲーム」** が近いかもしれません。

ゲームだと、ボス戦の前にセーブしておいて、
負けたらそこからやり直せますよね。

Gitも同じで、
「この状態を記録しておこう」と思ったタイミングで
自分でセーブポイントを作れるんです🥰


## 最初の準備

### パソコンにGitを入れる

まずはGitをパソコンに入れるところからですね🌸

パソコンの種類によって、やり方が少し違います。

**Macをお使いの方：**

ターミナル（Macに最初から入っている、文字で操作する画面です）を開いて、
以下の文字を打ち込んで Enter を押します。

```
brew install git
```

これは「brew（ブリュー）」という、
Macにソフトをインストールするためのツールを使った方法です🤭

**Windowsをお使いの方：**

「Git for Windows」というソフトをインターネットからダウンロードして、
画面の指示に従ってインストールしてください✨

**Linuxをお使いの方：**

ターミナルで以下を実行します。

```
sudo apt install git
```

ちゃんと入ったかどうかは、こう打てば確認できます。

```
git --version
```

`git version 2.xx.x` のような数字が表示されたら、成功です🥰

### 自分の名前を教えてあげる

Gitを使う前に、
「自分が誰か」を設定しておきます。

```
git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"
```

`config`（コンフィグ）は「設定」という意味です。
`--global` は「このパソコン全体で使う設定にしてね」というオプション🌸

これは「この変更は誰がやったのか」を記録するためのもの。
グループで使うときに大事になってきます。

ちゃんと設定できたか確認するには、

```
git config --list
```

これで名前とメールアドレスが表示されればOKです✨


## Gitの基本の流れ

ここが一番大事なところかもしれません。

でも安心してください。
やることは、たった **3ステップ** です😇


### ステップ① 「リポジトリ」を作る

まず出てくるのが **「リポジトリ（Repository）」** という言葉。

ちょっと構えてしまいますよね💦

でも要するに、
**「このフォルダの変更履歴を記録してね」とGitにお願いすること** です。

日記帳を1冊用意するようなイメージですね。
「このノートに、今日からの出来事を書いていくよ」と宣言する感じ🌸

具体的にはこうします。

```
mkdir my-project
cd my-project
git init
```

ひとつずつ説明しますね🤭

- `mkdir my-project` … 「my-project」という名前のフォルダを新しく作る（mkdirは「make directory＝フォルダを作る」の略です）
- `cd my-project` … そのフォルダの中に移動する（cdは「change directory＝場所を変える」の略です）
- `git init` … このフォルダでGitの記録を始める（initは「initialize＝初期化する」の略です）

これだけで、そのフォルダがGitの管理下に入ります✨

すでに誰かが作ったプロジェクトを取ってきたいときは、

```
git clone https://github.com/user/repository.git
```

こちらを使います。
`clone`（クローン）はその名のとおり「複製する」という意味で、
インターネット上にあるプロジェクトをまるごとコピーしてくれます🌸


### ステップ② 変更を記録する

ここ、最初はちょっとだけ戸惑うかもしれません🤭

Gitでは、変更を保存するまでに
**3つの場所** を通るんです。

ちょっとわかりにくいので、
**お弁当作り** にたとえてみますね🥰

**1つ目：作業ディレクトリ（キッチン）**
ここは普通にファイルを編集している場所です。
お弁当で言うと、材料を切ったり焼いたりしているキッチン。

**2つ目：ステージングエリア（お弁当箱に詰める台）**
記録したい変更を「次に保存するよ」と選んで置いておく場所です。
おかずが何品かできたけど、全部じゃなくて「今日のお弁当にはこれとこれを入れよう」と選ぶ段階。

**3つ目：リポジトリ（完成したお弁当）**
変更が確定して、履歴として記録される場所です。
フタを閉めて「今日のお弁当、完成！」となった状態。

なぜ真ん中の段階（ステージングエリア）があるかというと、
「変更したファイル全部じゃなくて、一部だけ記録したい」
というときに便利だからなんですよね🌸

具体的にはこうします。

```
git status
```

まず `status`（ステータス＝状態）で、
今どのファイルが変更されているかを確認します。

```
git add ファイル名
```

`add`（アド＝追加する）で、
記録したいファイルをステージングエリア（お弁当箱に詰める台）に置きます。

全部まとめて置きたいときは、

```
git add .
```

ピリオド（`.`）は「このフォルダの中の全部」という意味です🤭

そして、確定。

```
git commit -m "ここに変更内容のメモを書く"
```

`commit`（コミット＝確定する）で、ステージングエリアにある変更をまとめて記録します。

`-m` のあとに書く文章は **「コミットメッセージ」** と呼ばれるメモです。
**何を、なぜ変えたか** を短く書いておくと、
あとで振り返るときに本当に助かります✨

たとえばこんな感じ。

```
git commit -m "プロフィール画像を変更した"
git commit -m "お問い合わせページを追加した"
```


### ステップ③ 履歴を振り返る

記録した変更は、いつでも見返せます。

```
git log
```

`log`（ログ＝記録）で、
過去のコミット（セーブポイント）が新しい順に表示されます。

もう少しコンパクトに見たいときは、

```
git log --oneline
```

`--oneline` は「1つのコミットを1行で表示してね」というオプションです🌸

「前回の保存から、今どこを変えたんだっけ？」と確認したいときは、

```
git diff
```

`diff`（ディフ＝差分）で、
まだ記録していない変更の内容を見ることができます。

「あのとき何を変えたんだっけ？」
がすぐにわかる安心感、
一度味わうと手放せなくなるかもしれません🥰


## 「ブランチ」という考え方

少し進んだ話になりますが、
Gitの一番おもしろいところかもしれません😇

**ブランチ（Branch＝枝）** は、
**「今の状態をそのまま残しつつ、別の作業を試す」** ための機能です。

たとえるなら、
ノートのコピーを作って、そっちで下書きを試すような感じ🌸

元のノートはそのまま残っているから、
下書きがうまくいけば元のノートに反映するし、
気に入らなければ下書きだけ捨てればいい。

本体は無傷のまま、安心して実験できる。
これがブランチのいいところだと思っています✨

具体的にはこうします。

```
git branch
```

`branch` で、今あるブランチの一覧を確認できます。

```
git switch -c feature/new-function
```

`switch -c`（スイッチ＝切り替える、-c＝create＝新しく作る）で、
新しいブランチを作って、そこに移動します。

`feature/new-function` はブランチの名前で、
「何の作業をするブランチか」がわかるようにつけるのがコツです🤭

元のブランチに戻りたいときは、

```
git switch main
```

新しいブランチでの作業がうまくいって、
元のブランチに取り込みたくなったら、

```
git merge feature/new-function
```

`merge`（マージ＝統合する）で、
ブランチの内容を合流させることができます🌸

もし「やっぱり違ったな」と思えば、

```
git branch -d feature/new-function
```

`-d`（delete＝削除する）で、
ブランチごと消してしまえばOKです✨


## 「リモートリポジトリ」── 誰かとつながる

ここまでは、
全部自分のパソコンの中だけの話でした。

でも **「リモートリポジトリ」** を使うと、
自分の変更をインターネット上に保管したり、
他の人と共同作業したりできるようになります🥰

「リモート（Remote）」は「離れた場所にある」という意味。
つまり、自分のパソコンの外にある保管場所のことです。

代表的なサービスとして **GitHub（ギットハブ）** があります。
Gitで管理しているファイルを、
クラウド上に保存・共有できる場所だと思ってもらえたら🌸

```
git remote add origin https://github.com/user/repository.git
```

これは「インターネット上のこの場所を、保管先として登録するよ」という意味です。
`origin`（オリジン）は保管先につける名前で、慣習的にこう呼ばれています🤭

```
git push origin main
```

`push`（プッシュ＝押し出す）で、
自分のパソコンの変更をリモートに送ります。

```
git pull origin main
```

`pull`（プル＝引っ張る）で、
リモートにある他の人の変更を自分のパソコンに取り込みます。

`push` で送って、`pull` で受け取る。

この2つを覚えておけば、
チームでの共同作業の第一歩は踏み出せます🌸


## コマンドまとめ

一度、整理してみますね🤭

**はじめの一歩**
- `git init` … フォルダをGitの管理下にする
- `git clone <URL>` … インターネット上のプロジェクトをコピーする

**日常的に使うもの**
- `git status` … 今の状態を確認する
- `git add <ファイル名>` … 記録したい変更をステージングエリアに置く
- `git commit -m "メッセージ"` … 変更を確定して記録する（セーブポイントを作る）

**履歴を見る**
- `git log` … これまでの記録を振り返る
- `git diff` … まだ記録していない変更の差分を見る

**ブランチ（枝分かれ）**
- `git branch` … ブランチの一覧を見る
- `git switch <ブランチ名>` … ブランチを切り替える
- `git merge <ブランチ名>` … ブランチを統合する

**リモート（インターネット上の保管場所）**
- `git push` … 自分の変更をリモートに送る
- `git pull` … リモートの変更を自分のパソコンに取り込む

全部を一度に覚えなくても大丈夫です。
`add` → `commit` だけでも、十分Gitを使っていると言えます✨


## .gitignore ── 「記録しなくていいもの」を伝える

プロジェクトの中には、
Gitで管理しなくていいファイルもあります。

たとえば、
パスワードが書いてあるファイルとか、
ソフトが自動で作るファイルとか。

そういうものは **`.gitignore`（ギットイグノア）** というファイルに書いておくと、
Gitが「このファイルは記録しなくていいんだな」と無視してくれます🌸

`ignore` は「無視する」という意味ですね🤭

```
# パスワードなどが入りがちなファイル
.env

# ソフトが自動でダウンロードする部品（いつでも再取得できる）
node_modules/

# ソフトが自動で作る成果物
dist/
build/

# パソコンのOSが勝手に作るファイル
.DS_Store
Thumbs.db
```

ちなみに `#` で始まる行はコメント（メモ書き）で、
Gitには無視されます。

「記録するもの」と「しないもの」を分けておくのも、
きれいな管理の第一歩だったりします😇
