# Gitって何だろう？ ── 怖くない、はじめの一歩

## はじめに

ちょっと考えすぎかもしれないんですが…🤭

「Git」って名前はよく聞くけれど、
実際に触ろうとすると、
なんだか難しそうで手が止まってしまう。

そんな経験、ありませんか？

私もそうでした💦

黒い画面にコマンドを打つ。
「コミット」「ブランチ」「マージ」。
横文字がたくさん並んでいて、
正直、最初はちょっと怖かったです。

でも実は、
Gitがやっていることって、すごくシンプルなんですよね🌸

今日はそのあたりを、
一緒にゆっくり整理してみたいなと思います😇


## そもそも、Gitって何をしてくれるの？

少しだけ真面目な話をするとですね🤭

Gitは「**バージョン管理システム**」と呼ばれるものです。

……って言われても、ピンとこないですよね💦

もう少しやわらかく言うと、
**ファイルの「変更の歴史」を記録してくれるしくみ** です✨

たとえば、こんな経験ないでしょうか。

- レポートを書いていて、「やっぱり昨日の版に戻したい」と思った
- 複数人で同じファイルを編集していて、誰かの変更が消えてしまった
- `レポート_最終版_本当の最終版_v3.docx` みたいなファイルが増えていった

Gitを使うと、
こういった悩みから解放されます。

いつでも過去の状態に戻れるし、
誰がいつ何を変えたかも全部わかる。

「セーブポイントを自由に作れるゲーム」
みたいなものだと思ってもらえると、近いかもしれません🥰


## 最初の準備

### インストール

まずはGitをパソコンに入れるところからですね🌸

**macOSの方：**
```bash
brew install git
```

**Ubuntu / Debianの方：**
```bash
sudo apt install git
```

**Windowsの方：**
Git for Windows というツールをインストールしてください✨

入ったかどうかは、これで確認できます。

```bash
git --version
```

バージョン番号が表示されたら、大丈夫です🥰

### 自分の名前を教えてあげる

Gitを使う前に、
「自分が誰か」を設定しておきます。

```bash
git config --global user.name "あなたの名前"
git config --global user.email "your-email@example.com"
```

これは「この変更は誰がやったのか」を記録するためのもの。
チームで使うときに大事になってきます🌸

ちゃんと設定できたか確認するには、

```bash
git config --list
```

これで名前とメールが出てくればOKです✨


## Gitの基本の流れ

ここが一番大事なところかもしれません。

でも安心してください。
やることは、たった **3ステップ** です😇

### ステップ① リポジトリを作る

「リポジトリ」って言葉、ちょっと構えてしまいますよね💦

でも要するに、
**「このフォルダの変更履歴を記録してね」とGitに伝えること** です。

```bash
mkdir my-project
cd my-project
git init
```

これだけで、
そのフォルダがGitの管理下に入ります🌸

すでに誰かが作ったプロジェクトを取ってきたいときは、

```bash
git clone https://github.com/user/repository.git
```

こちらを使います✨

### ステップ② 変更を記録する（add → commit）

ここ、最初はちょっとだけ戸惑うかもしれません🤭

Gitでは、変更を保存するまでに
**3つの場所** を通るんです。

```
作業ディレクトリ  →  ステージングエリア  →  リポジトリ
 （ファイルを編集）   （記録する変更を選ぶ）   （変更を確定する）
```

うまく分解できているか自信はないですが💦
イメージとしては、

1. **作業ディレクトリ** … 普通にファイルを編集している場所
2. **ステージングエリア** … 「次に記録したい変更」を置いておく控室
3. **リポジトリ** … 実際に履歴として確定される場所

なぜ控室（ステージングエリア）があるかというと、
「変更したファイル全部じゃなくて、一部だけ記録したい」
というときに便利だからなんですよね🌸

具体的にはこうします。

```bash
# 今の状態を確認する
git status

# 記録したいファイルをステージングエリアに置く
git add ファイル名

# まとめて全部置くこともできる
git add .
```

そして、確定。

```bash
git commit -m "ここに変更内容のメモを書く"
```

この「メモ」がコミットメッセージです。

**何を、なぜ変えたか** を短く書いておくと、
あとで振り返るときに本当に助かります✨

### ステップ③ 履歴を振り返る

記録した変更は、いつでも見返せます。

```bash
# 履歴を見る
git log

# コンパクトに1行ずつ見る
git log --oneline

# 今の変更内容を確認する
git diff
```

「あのとき何を変えたんだっけ？」
がすぐにわかる安心感、
一度味わうと手放せなくなるかもしれません🥰


## ブランチという考え方

少し進んだ話になりますが、
Gitの一番おもしろいところかもしれません😇

**ブランチ** は、
「今の状態をそのまま残しつつ、別の作業を試す」ための機能です。

たとえるなら、
本の途中で「もしこの展開だったら？」という
もうひとつの物語を分岐させる感じ🌸

```bash
# 今あるブランチを確認する
git branch

# 新しいブランチを作って、そこに移動する
git switch -c feature/new-function

# 元のブランチに戻る
git switch main
```

新しいブランチで試した結果がよければ、
元のブランチに統合（マージ）できます。

```bash
git merge feature/new-function
```

もし「やっぱり違ったな」と思えば、
ブランチごと消してしまえばいい。

```bash
git branch -d feature/new-function
```

**本体は無傷のまま、安心して実験できる。**

これがブランチのいいところだと思っています✨


## リモートリポジトリ ── 誰かとつながる

ここまでは、
全部自分のパソコンの中だけの話でした。

でも、GitHubなどの「リモートリポジトリ」を使うと、
自分の変更をインターネット上に保管したり、
他の人と共同作業したりできるようになります🥰

```bash
# リモートリポジトリを登録する
git remote add origin https://github.com/user/repository.git

# 自分の変更を送る
git push origin main

# 他の人の変更を受け取る
git pull origin main
```

`push` で送って、 `pull` で受け取る。

この2つを覚えておけば、
チーム開発の第一歩は踏み出せます🌸


## コマンドまとめ

一度、整理してみますね🤭

| コマンド | やってくれること |
|---|---|
| `git init` | フォルダをGit管理下にする |
| `git clone <URL>` | 既存のリポジトリをコピーする |
| `git status` | 今の状態を確認する |
| `git add <ファイル>` | 変更をステージングエリアに置く |
| `git commit -m "メッセージ"` | 変更を確定して記録する |
| `git log` | 履歴を見る |
| `git diff` | 差分を確認する |
| `git branch` | ブランチの一覧を見る |
| `git switch <ブランチ名>` | ブランチを切り替える |
| `git merge <ブランチ名>` | ブランチを統合する |
| `git push` | リモートに送る |
| `git pull` | リモートから受け取る |

全部を一度に覚えなくても大丈夫です。
`add` → `commit` だけでも、十分Gitを使っていると言えます✨


## .gitignore ── 記録しなくていいものを伝える

プロジェクトの中には、
Gitで管理しなくていいファイルもあります。

パスワードが入ったファイルとか、
自動生成されるファイルとか。

そういうものは `.gitignore` というファイルに書いておくと、
Gitが無視してくれます🌸

```
# 環境変数（パスワードなどが入りがち）
.env

# 依存パッケージ（いつでも再取得できる）
node_modules/

# ビルドの成果物
dist/
build/

# OSが勝手に作るファイル
.DS_Store
Thumbs.db
```

「記録するもの」と「しないもの」を分けておくのも、
きれいな管理の第一歩だったりします😇
